# docker/

Local development stack and CI smoke infrastructure for `rsky-pds`.

## Stack overview

| Service | Image | Port | Purpose |
|---|---|---|---|
| `postgres` | `postgres:16-alpine` | 5432 | Primary database for rsky-pds |
| `minio` | `minio/minio:latest` | 9000 / 9001 | S3-compatible blob storage (UI at 9001) |
| `minio-setup` | `minio/mc:latest` | — | One-shot: creates the blob bucket on first start |
| `rsky-pds` | `rsky-pds-dev:latest` | 2583 | The PDS binary |
| `plc` *(CI only)* | `plc-dev:latest` | 2582 | Local PLC directory — avoids calling `plc.directory` during tests |
| `migrator` *(CI only)* | `postgres:16-alpine` | — | One-shot: runs Diesel SQL migrations without `diesel-cli` |

## Local development

### Prerequisites

- A container runtime with Compose support (Docker Desktop, Colima, Rancher Desktop, or Podman)
- `openssl`, `curl`, `jq` in PATH
- `cargo` + `diesel-cli` (installed automatically by the setup script if absent)
- macOS: `brew install libpq` before the first run

### First-time setup

```bash
bash scripts/setup-local.sh
```

This script:
1. Generates `docker/.env` from `docker/.env.example` (secp256k1 keys, admin password)
2. Starts postgres + minio and waits for them to be healthy
3. Runs Diesel migrations via `diesel-cli` on the host
4. Builds and starts `rsky-pds` (first build can take several minutes)
5. Creates two test accounts (`alice.test` / `bob.test`)

### Daily use

```bash
# Bring the stack up
docker compose -f docker/docker-compose.yml --env-file docker/.env up -d

# Verify endpoints
bash scripts/verify-stack.sh

# Tear down (keeps volumes)
docker compose -f docker/docker-compose.yml --env-file docker/.env down

# Tear down and wipe data
docker compose -f docker/docker-compose.yml --env-file docker/.env down -v
```

### Configuration

Edit `docker/.env` (generated by `setup-local.sh`, gitignored). Key variables:

| Variable | Default | Notes |
|---|---|---|
| `PDS_HOSTNAME` | `localhost` | Determines accepted handle domain |
| `PDS_SERVICE_DID` | `did:web:localhost` | Required — server panics if absent |
| `PDS_ADMIN_PASS` | `hunter2` | HTTP Basic auth for admin endpoints |
| `PDS_INVITE_REQUIRED` | `false` | Set `true` to require invite codes |
| `AWS_ENDPOINT_BUCKET` | `rsky-pds-blobs` | MinIO bucket name |

Database and object storage URLs are overridden by `docker-compose.yml` to point at the compose-internal service names; the values in `.env` are used when running the binary directly on the host.

## CI overlay (`docker-compose.ci.yml`)

The CI pipeline uses both files together:

```
docker compose \
  -f docker/docker-compose.yml \
  -f docker/docker-compose.ci.yml \
  --env-file docker/.env \
  up -d --no-build
```

The overlay adds:

- **`plc`** — `@did-plc/server` running against a local PostgreSQL database. Allows `createAccount` to anchor `did:plc` operations without network access. PDS is pointed at it via `PDS_DID_PLC_URL=http://plc:2582`.
- **`migrator`** — runs the Diesel SQL migrations directly via `psql`, so CI does not need `diesel-cli`. Also creates the `plc` database for the PLC service.

`rsky-pds` in CI `depends_on` both, so startup order is guaranteed:
`postgres` → `migrator` → `plc` (healthy) → `rsky-pds`

## File inventory

| File | Purpose |
|---|---|
| `docker-compose.yml` | Base stack: postgres, minio, rsky-pds |
| `docker-compose.ci.yml` | CI overlay: adds plc + migrator, overrides PDS_DID_PLC_URL |
| `.env.example` | Template for `docker/.env` — commit safe (no real secrets) |
| `Dockerfile.dev` | Development image for rsky-pds (full workspace build inside Docker) |
| `Dockerfile.runtime` | Minimal runtime image for CI — copies pre-built binary from the host |
| `Dockerfile.plc` | Pre-bakes `npm install @did-plc/server` to avoid runtime installs |

### `Dockerfile.runtime` vs `Dockerfile.dev`

`Dockerfile.dev` does a full `cargo build` inside the image (slow, suitable for local dev where you want self-contained Docker builds). `Dockerfile.runtime` expects the binary to already exist at `target/release/rsky-pds` and just copies it in — this is what CI uses after compiling on the runner with Swatinem's cache, keeping image build time under 30 seconds.

Base image is `ubuntu:24.04` to match the GLIBC version on `ubuntu-latest` runners (2.39). `debian:bookworm-slim` only ships GLIBC 2.36 and will produce "version not found" errors at startup.

## CI smoke pipeline (`.github/workflows/smoke.yml`)

Three jobs run in sequence after a parallel build phase:

```
test ─┐
      ├─→ smoke ──→ atproto-smoke
build ┘
```

- **`test`** — `cargo test -p rsky-pds` (debug build, fast). Runs in parallel with `build`.
- **`build`** — compiles the release binary on the runner, assembles `rsky-pds-dev` and `plc-dev` images, uploads them as artifacts. Uses `CARGO_PROFILE_RELEASE_OPT_LEVEL=1` and `CARGO_PROFILE_RELEASE_INCREMENTAL=true` to keep compile time under 5 minutes in CI without touching `Cargo.toml`.
- **`smoke`** — loads the pre-built images, starts the full CI stack, waits for rsky-pds, then runs `scripts/verify-stack.sh`.
- **`atproto-smoke`** — runs on push (all branches) or PRs labeled `smoke`. Clones [`aliceisjustplaying/atproto-smoke`](https://github.com/aliceisjustplaying/atproto-smoke) and exercises the PDS with real account operations via `bunx tsx bin/atproto-smoke.ts validate --mode dual`.

## Known limitations

These are tracked in `.claude/TASKS.md` and have not yet been fixed upstream:

- **`uploadBlob` returns 500** — the AWS SDK client is built without `force_path_style = true`. MinIO requires path-style URLs; without it the SDK constructs virtual-hosted URLs (`bucket.minio:9000`) that don't resolve inside Docker. (`rsky-pds/src/lib.rs:226`)
- **`createInviteCode` unauthenticated returns 500** — `AdminToken::from_request` returns `Status::BadRequest` for missing auth but Rocket surfaces it as 500. (`rsky-pds/src/auth_verifier.rs:635`)
- **`PDS_SERVICE_DID` panic** — `describe_server.rs:18` calls `.unwrap()` on the env var. The var is set in `.env.example`; removing it causes a startup panic rather than a clean error.
